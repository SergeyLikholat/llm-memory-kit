# Как работает память — подробно, простыми словами

## Три уровня: карта → индекс → детали

Память построена как **картотека с оглавлением**, чтобы не забивать контекст:

```
УРОВЕНЬ 0 — Главный путеводитель   (всегда в контексте, ~50 строк)
  список проектов + «за чем сюда идти» + указатель вниз
      ↓ по запросу про проект
УРОВЕНЬ 1 — Оглавление проекта      (подтягивается при работе по теме)
  список тем проекта + указатели на карточки
      ↓ по запросу про конкретную тему
УРОВЕНЬ 2 — Карточка темы (шард)    (грузится только нужная)
  накопленное знание: решения, факты, статусы, открытые вопросы
```

**Накопление** идёт на уровне 2 (карточки растут), а карты (0–1) остаются тонкими:
новая тема за год = +1 строка в оглавлении, а не страница текста. Поэтому память
копится без ограничения по объёму, но контекст не раздувается.

## Ночной поток

```
1. digest   — читает РОДНЫЕ транскрипты сессий (~/.claude/projects/*/*.jsonl),
              инкрементально по mtime (обрабатывает только изменённые), дедуп по
              sessionId; группирует по проектам (Telegram — по топику; иначе — буфер)
2. classify — раскидывает общий буфер по проектам (Claude решает по «purpose», батчами)
3. builder  — двухфазно: маршрутизирует новые записи по темам, затем точечно
              обновляет ТОЛЬКО затронутые темы (не перегенерирует все разом)
```

Всё работает на **вашей подписке Claude** (headless-вызовы `claude -p`, модель из
`MEMKIT_MODEL`, по умолчанию sonnet — экономнее), без API-ключей.

**Экономия и защита от сжигания лимита.** builder обрабатывает только контуры со
свежими данными (сравнивает время буфера и индекса). Прогон идёт в окнах 02:30 / 09:00
/ 16:00, а не каждые полчаса. Обёртки держат гард: успех фиксируется раз в сутки,
максимум 3 попытки, лимит подписки (`exit 42`) → попытка в следующее окно, ошибка кода
→ ретраи останавливаются до починки. Файлы при этом не портятся.

**Слияние дублей.** Со временем близкие темы плодятся. `consolidate.js` группирует их
через LLM и сливает в один шард (с бэкапом). `builder.js --reindex` пересобирает все
оглавления без единого вызова LLM — мгновенно и бесплатно.

## Авто-извлечение (два хука)

- **При старте сессии** — подставляется главный путеводитель.
- **На каждый запрос** — релевантные темы находятся ранжированием: слово в теге/заголовке
  карты («тема ПРО это») весит кратно больше случайного упоминания в теле, редкие слова
  тащат сильнее частых. Для верных попаданий в подсказку подставляется **сам факт** —
  строки шарда со словами запроса, а не только ссылка на файл. Болтовня («привет»,
  «спасибо») не срабатывает — контекст не засоряется. Приватные контуры телом не
  индексируются (только указатель).

## Определение проекта

- **Telegram**: по топику (если настроен routing) — точно.
- **Работа в папке проекта**: по рабочей директории.
- **Иначе** (например, всегда в корне): Claude **классифицирует по смыслу**, опираясь
  на «purpose» проектов. Не уверен → кладёт в общий контур (не гадает).

## Граф сущностей (горизонтальные связи)

Дерево (index → проект → тема) хорошо для навигации, но не показывает, что
**одна сущность связывает разные проекты**. Для этого — слой хабов:

```
entities/<сущность>.md   — узел-хаб
  ## В памяти сессий:  ссылки на темы разных проектов
  ## В Obsidian:       ссылки на заметки/карты
```

Ночью (после сборки памяти и Obsidian-sync) LLM выделяет **именованные сущности**
(люди, объекты, организации, ключевые инструменты), встречающиеся в 2+ местах, и
строит для них хабы. Абстрактные темы не выделяются — они покрыты тегами.

Хаб всплывает в подсказке через тот же retrieval-хук: упомянул сущность —
получил узел со всеми связанными местами разом, из обеих баз. Шарды при этом
остаются чистыми (связи живут в отдельных файлах-хабах, а не внутри тем).
